# Hoop 核心主流程重设计

## 目标

把当前「首页概览 + 训练上传 + AI 单独页」的分散结构，重构为一条明确的主流程：

`拍视频 -> 上传 -> 进入视频动态 -> 手动触发当前视频 AI 分析 -> 结果回到同一条视频动态 -> 按日期持续沉淀`

产品中心从「任务卡片」切换为「视频动态流」。

---

## 先说结论

当前实现已经有本地用户、视频上传、上传记录这些基础能力，但核心问题不是功能缺失，而是主链路被拆散了：

- 首页是静态摘要，不是内容入口
- 训练页承担上传，但不是视频主时间线
- AI 页是独立静态页面，没有附着到具体视频
- “按天”只体现在局部查询，不是完整的时间线结构

这会导致用户完成一次上传后，看不到一个自然闭环，也很难形成“像朋友圈一样持续积累视频内容”的使用习惯。

---

## 现状问题

### 0. 当前页面存在较多预设演示数据，不适合真实新用户首屏

现在的页面内容里混有不少“演示型文案”和“预设统计”：

- 连续打卡
- 成长值
- 最近一次 AI 分析
- 本周观察重点
- 默认训练任务完成情况

这些内容在新用户首次进入时实际上都不存在。

如果继续保留这类预设内容，会带来几个问题：

- 新用户会误以为系统已经有自己的历史数据
- 产品会显得不真实，降低信任感
- 后续真实数据接入后，页面结构会反复返工

结论：新用户默认状态必须是零数据状态，不能依赖写死的演示数据撑页面。

### 1. 当前核心实体仍然是“上传记录”，不是“视频动态”

现有 `UploadedTrainingVideoRecord` 更像上传日志，只够支撑：

- 文件路径
- 上传时间
- 球员归属
- 当天筛选

它还不能支撑真正的视频主流程，因为缺少这些关键语义：

- 训练主题或动作标签
- 视频封面或展示摘要
- AI 分析状态
- AI 分析结果摘要
- 对比基线视频
- 是否需要重点关注

结论：如果核心实体还是“上传记录”，最后做出来的会是“训练后台列表”，不是“视频动态流”。

### 2. 首页没有成为真正的内容入口

如果目标是类似微信朋友圈，那么首页不应该先展示大段静态概览，再把视频放到别的 Tab 里。

首页应该直接展示：

- 最近日期分组
- 每个日期下的视频动态
- 每条视频上的 AI 结果摘要
- 明显的拍摄/上传入口

首页的职责应该从“概览”改为“视频流”。

### 3. AI 分析被放在独立页面，打断了用户心流

对用户来说，AI 分析不是一个独立目的地，而是某条视频上传后的自然结果。

更合理的关系应该是：

- 视频是主内容
- AI 分析是视频卡片上的附着信息
- 点开视频详情后，可以看完整 AI 复盘
- 历史对比也应该围绕某条视频展开

如果 AI 继续单独放一个 Tab，它很容易长期停留在展示页，而不是高频使用主流程的一部分。

### 4. “按天分组”定义了数据筛选，但没有定义产品时间线

“按日期分组”不能只理解成：

- 查询今天的数据
- 管理员看今天所有人

真正的产品定义应该是：

- 页面天然是时间线
- 顶层 section 是日期
- 日期下挂视频动态
- 默认展示今天、昨天、近几天
- 继续下滑可以看到更早日期

否则实现仍然会停留在“今日上传”模块，而不是内容流。

### 5. 家长端与球员端没有走同一条内容主线

当前设计更像：

- 球员端负责上传
- 家长端负责总览和管理

但如果产品核心是视频动态，那么家长端和球员端应该共享同一种信息结构，只是权限和视角不同：

- 球员端看自己的视频时间线
- 家长端看家庭成员的视频时间线
- 家长端可按成员过滤
- 两端都围绕“日期 -> 视频 -> AI 反馈”组织

这样整个产品会更统一。

## 重设计原则

### 1. 视频是第一内容单元

每次训练的主产物不是“完成了一项任务”，而是“生成了一条可回看的训练视频动态”。

### 2. 日期是第一组织维度

先按日期分组，再在日期下展示视频，而不是先按页面功能分类。

### 3. AI 是附着能力，不是独立岛屿

AI 分析要跟着视频走，不应该独立悬空。

### 4. 上传成功不等于流程结束

上传只是中间态，完整闭环必须包含：

- 上传成功
- 用户对当前视频手动发起 AI 分析
- AI 输出结果
- 结果回到原视频卡片

### 5. 家长和球员共享同一套内容骨架

差异只体现在：

- 看自己的流还是全家流
- 是否能上传
- 是否能管理成员

而不是完全不同的信息架构。

### 6. 零数据状态必须成为第一公民

这次重构不能默认假设用户已经有：

- 历史视频
- AI 分析结果
- 连续打卡记录
- 周进步趋势
- 已完成训练任务

系统应该优先围绕以下真实场景设计：

- 新建成员后第一次进入
- 当天还没有任何视频
- 上传过视频但还没做 AI 分析
- 家长端当前没有任何成员上传

如果这些状态设计清楚了，真实数据充足后的界面自然会成立；反过来则很容易变成样板页面。

---

## 新的核心信息架构

建议把主壳层收敛为下面这套结构：

- `动态`
- `我的`

其中 `动态` 是新的默认首页，也是唯一主内容页。

这意味着：

- 原 `Home`、`Training`、`AI` 的核心职责并入 `动态`
- `比赛` 不再作为一级 Tab，而是作为视频内容类型并入 `动态`
- `AI` 不再作为一级 Tab，而是作为当前视频上的手动分析能力存在

## 核心实体建议

### 1. `TrainingVideoPost`

建议新增一个面向展示层和业务层的核心实体，名字可以是 `TrainingVideoPost`、`TrainingVideoEntry` 或 `VideoMoment`，重点不是命名，而是语义升级。

建议字段至少包括：

```swift
@Model
final class TrainingVideoPost {
    @Attribute(.unique) var id: String
    var playerID: String
    var objectKey: String
    var fileName: String
    var createdDay: Date
    var uploadedAt: Date
    var fileSize: Int64

    var drillName: String?
    var note: String?
    var analysisStatusRawValue: String
    var latestAnalysisSummary: String?
    var latestRecommendation: String?
    var baselineVideoPostID: String?
}
```

最低要求不是一次把所有字段都做完，而是先把模型从“上传日志”升级到“视频动态”。

### 2. `VideoAnalysisRecord`

如果后续 AI 分析会重跑，建议把 AI 结果独立建模，而不是直接把所有字段拍平到视频表里。

```swift
@Model
final class VideoAnalysisRecord {
    @Attribute(.unique) var id: String
    var videoPostID: String
    var statusRawValue: String
    var summary: String?
    var recommendation: String?
    var comparedVideoPostID: String?
    var createdAt: Date
    var finishedAt: Date?
}
```

这样可以支撑：

- 分析中
- 分析完成
- 分析失败
- 重新分析
- 历史对比

如果第一阶段想控复杂度，也可以先只在 `TrainingVideoPost` 上挂一份最新结果，但文档里要先承认这是阶段方案。

## 主流程重排

## 核心操作链路

这一版产品只保留两条真正高频的核心链路：

- 球员生产内容
- 家长查看与跟进内容

所有页面和状态都应该围绕这两条链路服务，而不是围绕演示卡片或分散的功能页服务。

### 链路 A：球员首次进入

1. 进入 App，完成设备解锁与身份选择。
2. 进入 `动态` 页。
3. 如果当前没有任何视频，显示零数据空状态。
4. 页面明确告诉用户：
   - 这里会按日期沉淀训练/比赛视频
   - 第一步是上传第一条视频
   - AI 分析需要针对当前视频手动触发
5. 用户点击“上传第一条视频”或“拍训练视频”。

这个链路的目标不是展示统计，而是尽快把用户带到第一条视频内容产生。

### 链路 B：球员上传一条视频

1. 用户在 `动态` 页点击上传或拍摄入口。
2. 选择一个训练视频或比赛视频。
3. 视频上传到 OSS。
4. 本地立即写入一条 `TrainingVideoPost`。
5. 这条视频出现在当天日期分组下。
6. 初始状态为：
   - 已上传
   - 待分析
7. 页面不自动触发 AI，不跳转到其他 Tab。

这个链路的目标是让“上传成功”立刻变成一条可见、可回看的内容。

### 链路 C：球员针对当前视频手动触发 AI 分析

1. 用户在某条视频卡片上点击 `开始 AI 分析`。
2. 或者进入视频详情页后点击 `开始 AI 分析`。
3. 系统只对当前这条视频发起分析。
4. 视频状态切换为 `processing`。
5. 分析完成后，结果写回当前视频。
6. 卡片和详情同步更新：
   - 一句话结论
   - 下一次训练重点
   - 如有需要，展示与上一条相似动作视频的对比
7. 用户之后可以再次点击 `重新分析`。

这个链路的关键是：AI 是针对某条视频的动作，不是自动后台流程，也不是独立页面流程。

### 链路 D：球员回看历史成长

1. 用户进入 `动态` 页。
2. 页面按日期分组展示历史视频。
3. 用户查看今天、昨天、更早日期的视频。
4. 每条视频都能看到：
   - 基本信息
   - 是否做过 AI 分析
   - 最新分析摘要
5. 用户点进详情，看单条视频的完整复盘。

这个链路的目标是形成连续的成长时间线，而不是一次性任务完成页。

### 链路 E：浏览视频内容

1. 用户进入 `动态` 页。
2. 页面默认按日期分组展示视频流。
3. 用户先定位某一天，再浏览该日期下的视频卡片。
4. 在卡片层，用户快速获取：
   - 这是训练还是比赛
   - 是谁的视频
   - 上传时间
   - 是否已经做过 AI 分析
   - 是否已有一句话分析摘要
5. 用户点击某条视频进入详情页。
6. 详情页展示：
   - 视频基本信息
   - 当前分析状态
   - 完整 AI 结果或待分析状态
   - 同一天其他相关视频

这个链路的重点是：先按日期找到视频，再围绕具体视频继续浏览，而不是先进入一个抽象功能页。

### 链路 F：分析一条已有视频

1. 用户在 `动态` 页或视频详情页看到某条视频。
2. 用户判断当前视频值得分析。
3. 点击 `开始 AI 分析`。
4. 系统只对当前这条视频发起分析。
5. 视频状态切换为 `processing`。
6. 分析完成后，结果回写当前视频：
   - 卡片更新摘要
   - 详情页更新完整结果
7. 用户继续查看：
   - 一句话结论
   - 下一次训练重点
   - 如有需要，展示与上一条相似动作视频的对比
8. 用户后续可以再次点击 `重新分析`。

这个链路的重点是：分析是浏览具体视频后的主动操作，不是上传后的自动动作，也不是单独页面中的独立任务。

### 链路 G：家长查看家庭动态

1. 家长身份进入 `动态` 页。
2. 页面按日期展示全家成员的视频动态。
3. 如果没有任何成员视频，展示家庭空状态：
   - 还没有训练动态
   - 等待成员上传第一条视频
   - 可进入成员管理
4. 如果已有内容，每条卡片显示：
   - 成员名字
   - 视频类型
   - 上传时间
   - AI 状态
   - 是否已有分析结论
5. 家长点进详情查看具体视频和 AI 结果。

这个链路的目标不是看抽象报表，而是直接围绕成员视频进行跟进。

### 链路 H：家长管理，不打断内容主链路

1. 家长从 `我的` 页进入成员管理。
2. 完成新增成员、编辑成员、切换身份等操作。
3. 操作完成后回到 `动态` 页继续看内容流。

也就是说，管理能力存在于 `我的`，但不会抢占内容主入口。

### 球员端

#### 1. 打开 App

进入默认首页，不是“今天任务概览”，而是“我的训练动态”：

- 如果已有数据：今天 / 昨天 / 更早日期
- 如果没有数据：展示零数据空状态

#### 2. 点击拍摄或上传

入口要固定出现在首页首屏，例如：

- 顶部主按钮“拍训练视频”
- 首个空状态 CTA“上传今天第一条视频”

#### 3. 上传后立即生成一条视频卡片

先进入流里，状态为：

- 上传中
- 待分析
- AI 分析完成

用户不需要跳去另一个 AI 页面找结果。

#### 4. 在当前视频上手动触发 AI 分析

每条视频卡片或详情页都应有明确操作：

- `开始 AI 分析`
- `重新分析`

分析动作必须绑定当前视频，而不是系统自动后台触发。

#### 5. AI 结果回写原卡片

视频卡片展示：

- 一句话结论
- 下一次训练重点
- 是否与上次同动作对比

#### 6. 点进视频详情

详情页展示：

- 视频基本信息
- AI 完整分析
- 与上一条相似动作视频的对比
- 同一天的其他视频

### 家长端

#### 1. 打开 App

进入“家庭训练动态”，同样按日期分组。

如果当前家庭还没有任何视频，首页应直接展示：

- 还没有训练动态
- 等待成员上传第一条视频
- 成员管理入口

#### 2. 浏览家庭成员视频

每条视频卡片带上：

- 成员头像/名字
- 上传时间
- AI 状态
- 是否值得跟进

#### 3. 进入详情

重点看：

- 这条视频的 AI 结论
- 相比上一次是否进步
- 是否需要提醒补拍或继续训练

## 页面建议

### 1. 动态首页

首页应由以下几个部分组成：

#### 顶部区域

- 当前身份
- 日期范围提示
- 主行动按钮“拍训练视频”

#### 主体内容

- 以日期为 section 的视频流
- 每个日期下按上传时间倒序
- 每条卡片展示视频缩略信息和 AI 摘要
- 视频类型可包含训练、比赛等不同来源

#### 空状态

- 今天还没有视频
- 引导拍第一条训练视频
- 不展示任何虚构的统计值或 AI 结果

#### 新用户首屏原则

新用户第一次进入时，首页应只回答三个问题：

- 这里将来会沉淀什么内容
- 我现在第一步该做什么
- AI 分析是上传后手动触发的

不要为了填满页面加入演示数据。

### 2. 视频卡片

每条卡片建议包含：

- 球员信息
- 上传时间
- 动作标题
- 视频占位封面
- AI 状态 badge
- AI 摘要
- 下一步建议
- `开始 AI 分析` 按钮

如果暂时还没有真实缩略图，也应该先有统一的视频卡片样式，不要继续复用普通列表行。

没有视频时，不显示卡片占位假数据，直接显示真实空状态。

### 3. 视频详情页

详情页才是 AI 结果完整承载页：

- 视频信息
- AI 总结
- 关键改进点
- 和上一条相似视频的差异
- 建议下次拍摄角度
- 手动触发分析或重新分析入口

### 4. 导航建议

建议最终主导航固定为：

- `动态`
- `我的`

对应职责：

- `动态`：按日期分组的视频流、上传入口、当前视频的 AI 分析入口
- `我的`：当前身份、成员切换、家长管理、设置

比赛内容不再单独作为一级导航，而是作为动态流中的视频类型存在。

## 日期分组规则

为了让“按日期展示”真正稳定，建议文档明确这些规则：

### 分组规则

- 一级分组键为 `createdDay`
- `createdDay` 统一归一化到本地自然日 00:00
- 列表展示时使用“今天 / 昨天 / 具体日期”的文案

### 排序规则

- 日期组按新到旧排序
- 日期组内视频按 `uploadedAt` 倒序

### 家长端聚合规则

- 先按日期分组
- 再在同一日期下混排所有成员的视频
- 卡片上明确标记成员身份

这样“日期”才是第一层级，“成员”是视频卡片上的属性，而不是反过来。

如果没有任何可分组数据，不要伪造“今天”分组，直接进入空状态。

## AI 分析链路建议

文档里建议把 AI 状态明确写出来，否则工程上容易继续做成假数据展示。

### 推荐状态

- `idle`
- `processing`
- `completed`
- `failed`

### 推荐链路

1. 选择视频
2. 上传 OSS
3. 写入本地 `TrainingVideoPost`
4. 初始状态标记为 `idle`
5. 用户在当前视频卡片或详情页点击“开始 AI 分析”
6. 状态切到 `processing`
7. 分析完成后更新卡片摘要和详情数据

### 触发原则

- AI 分析不是上传成功后的自动动作
- AI 分析必须由用户针对当前视频手动发起
- 同一条视频应支持重新分析
- 未触发分析的视频，仍然应正常出现在日期流中
- 没有分析结果时，不展示默认 AI 总结文案

### 对比分析建议

如果你的“AI 复制分析”本意是“模仿/复现/对比上一次动作表现”，建议在文档中明确：

- 对比对象优先选择最近一次相同动作视频
- 如果没有同动作视频，则退化成单条视频分析
- 对比分析只在用户手动触发当前视频分析时执行
- 首页卡片只展示一句对比结论
- 详情页再展示完整对比

## 与当前实现的对应关系

### 当前已具备

- 本地成员系统
- OSS 视频上传
- SwiftData 上传记录
- 按角色分视图

### 当前最需要调整

- `HomeView` 不能继续停留在静态概览
- `TrainingView` 不能只是“训练计划 + 上传入口”
- `AIAnalysisView` 不能继续是独立静态说明页
- `UploadedTrainingVideoRecord` 不能继续只做上传日志角色
- 各页面不能继续依赖预设演示数据来填充首屏
- 导航层不能继续把首页、训练、AI、比赛拆成多个并列主入口

### 零数据落地要求

第一阶段实现时，至少要保证：

- 新球员进入后首页为空状态，不显示假的“连续打卡第 8 天”
- 没有上传记录时，不显示假的“最近一次视频分析”
- 没有家庭数据时，家长端不显示假的成员进展结论
- AI 页面或 AI 区块在未触发时，只显示可操作入口，不显示伪造分析结果

## 推荐实施顺序

### 第一阶段：先把主入口改对

- 把 `Home` 改造成视频动态流
- 把“今天上传”扩展成“按日期分组的历史流”
- 保留现有上传逻辑，先把真实数据挂进新流里
- 先完成零数据空状态，移除预设演示文案
- 把原有多 Tab 结构收敛为 `动态 + 我的`

### 第二阶段：把 AI 挂回视频

- 给每条视频加 AI 状态
- 让 AI 结果显示在视频卡片和详情页
- 弱化独立 AI 页

### 第三阶段：做真正的对比分析

- 引入同动作基线视频
- 支持“和上一次相比”的结果表达
- 给家长端增加重点跟进标记

## 最终建议

这次重构不建议从“页面美化”开始，而应该先统一三件事：

- 核心实体是视频动态，不是上传记录
- 核心入口是视频流，不是概览卡片
- 核心闭环是上传后回到原视频看到 AI 结果

导航层也建议同步收敛为：

- `动态`
- `我的`

其中比赛作为内容类型并入动态流，而不是单独成为一级入口。

只要这三件事定下来，后面的 SwiftUI 拆分、Tab 调整、卡片样式、详情页设计都会顺下来。

如果后续继续推进实现，建议以这份文档为准，原 `user-system-design.md` 继续承担“本地用户系统与权限模型”职责，两份文档并行存在。
