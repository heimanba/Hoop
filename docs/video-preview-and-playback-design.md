# Hoop 视频首图与播放设计

## 文档目标

这份文档用于定义视频动态流里的“可视化视频展示能力”，补齐当前只有文本卡片、没有首图和播放体验的问题。

它关注四件事：

- 上传完成后，用户应该看到什么
- 动态卡片和详情页各自承担什么职责
- 需要补哪些数据字段和本地处理链路
- 按什么顺序落地，才能在每一步都保持可运行、可验证

---

## 问题背景

当前动态流已经打通了主链路：

`选视频 -> 上传到 OSS -> 写入 TrainingVideoPost -> 出现在当天分组 -> 可触发 AI 分析`

但现在的视频卡片仍然是“文本信息卡片”，主要展示：

- 文件名
- 上传时间
- 内容类型
- AI 状态
- 查看详情 / 开始分析

这会带来三个直接问题：

- 用户上传完成后，看不到熟悉的视频封面反馈，感知不够强
- 卡片无法承接“我刚上传的是哪条视频”的快速确认动作
- 详情页无法直接播放，用户仍然要跳出当前主链路才能回看视频

所以这次设计的目标不是做一个更漂亮的卡片，而是把“视频是内容主体”这件事真正落到界面和数据结构上。

## 设计目标

### 本轮必须完成

- 上传完成后，动态流卡片显示视频首图
- 点击卡片或详情入口后，可以在详情页直接播放视频
- 上传后首条反馈仍然保持在当前动态页，不跳走
- 首图加载失败时，界面要有稳定的降级表现

### 本轮不做

- 卡片内直接自动播放
- 卡片内联完整播放器
- 视频编辑、裁剪、封面自选
- 远程流媒体优化和复杂缓存策略
- 真实 AI 结果与时间轴标注联动

这次设计优先做“强确认感 + 低实现风险”的版本，也就是：

- 卡片展示静态首图
- 详情页承担正式播放

## 体验目标

### 1. 上传中

动态页顶部继续保留当前上传状态：

- 进度文案
- 进度条
- 上传按钮禁用

这部分不需要因为首图能力而重做。

### 2. 上传完成后

上传完成后，用户仍留在“动态”页，但当前页反馈要从“纯文本成功提示”升级为“看见自己的视频”。

此时页面应同时满足：

- 顶部主操作卡显示“训练已入流”或“比赛已入流”
- 当天分组最上方立即出现新视频卡片
- 新卡片顶部有视频首图
- 首图上有明显的播放提示图标
- 卡片下半部分继续展示内容类型、时间、AI 状态和操作按钮

设计原则：

- 用户第一眼先认出视频，再读文字
- 首图是主信息，标题和状态是辅助信息
- 卡片点击后默认进入详情页，而不是在列表里直接展开重播放器

### 3. 动态卡片

动态卡片的核心原则是：

- 首图是主视觉
- 信息是辅助理解
- 操作尽量后置到详情页

也就是说，卡片不应该因为接入了首图，就继续维持“两个大按钮 + 大段文案”的信息密度。视频内容应该先被看见，操作其次。

视频卡片建议分为上下两层：

- 上层：16:9 或接近 16:9 的首图区域
- 下层：标题、时间、状态、轻量操作

首图区域应该具备：

- 封面图
- 中央播放 icon
- 左下角内容类型 badge 或时长信息
- 加载失败时的稳定占位背景

文本信息区建议收敛到最少必要信息：

- 文件名或动作标题
- 上传时间与 AI 状态合并在一行
- 有 AI 摘要时只显示一行摘要
- 无 AI 摘要时不强行显示大段占位文案

卡片交互建议：

- 整张卡片可点击，进入详情页
- 不再保留一个独立的大号 `查看详情` 按钮
- `开始 AI 分析 / 重新分析` 改为轻量按钮或文本按钮
- 如果希望进一步克制首页操作，也可以将 AI 按钮仅保留在详情页

第一版建议采用折中方案：

- 整卡点击进入详情页
- 卡片上保留一个轻量 `分析` 操作
- 播放、完整 AI 信息和更多元数据全部放到详情页

### 4. 详情页

详情页需要从“信息页”升级为“视频详情页”。

推荐顺序：

1. 顶部播放器区域
2. 基础信息卡
3. AI 结果卡

详情页播放器行为：

- 默认显示首图和播放按钮
- 用户点击后开始播放
- 支持暂停 / 继续播放
- 支持基础系统控件
- 不要求第一版支持拖拽标注、倍速和片段剪辑

如果播放地址不可用，详情页应降级为：

- 仍显示首图
- 展示“暂时无法播放，请稍后重试”
- 不影响 AI 信息区展示

## 方案选择

### 方案 A：卡片首图 + 详情页播放

这是推荐方案。

优点：

- 与当前动态流结构最兼容
- 用户能立刻看到视频内容
- 列表性能和复杂度可控
- 避免在流里塞多个活跃播放器
- 便于后续扩展成详情页更多能力

代价：

- 用户需要多点一次进入详情页播放

### 方案 B：卡片内直接播放

不推荐作为第一版。

问题：

- 滚动列表里的播放器状态管理更复杂
- 容易引入性能和声音管理问题
- 多条视频混在时间流里时交互容易混乱

结论：

第一版采用“卡片首图 + 详情页播放”，后续如果用户有强需求，再评估卡片内轻量预览。

## 数据模型设计

当前 `TrainingVideoPost` 不足以支撑首图和播放，建议补充最小必要字段。

建议新增字段：

```swift
@Model
final class TrainingVideoPost {
    var remoteVideoURLString: String?
    var thumbnailURLString: String?
    var localThumbnailRelativePath: String?
    var durationSeconds: Double?
    var previewStatusRawValue: String
}
```

字段说明：

- `remoteVideoURLString`
  - 上传成功后的远程视频地址
  - 用于详情页远程播放
- `thumbnailURLString`
  - 预留给未来远程缩略图方案
  - 第一版可以先不写入真实值
- `localThumbnailRelativePath`
  - 本地生成并缓存的首图路径
  - 第一版卡片首图优先读取它
- `durationSeconds`
  - 用于卡片角标和详情页基础信息
- `previewStatusRawValue`
  - 用于表达预览资源状态
  - 建议枚举：`pending`、`ready`、`failed`

对应建议新增：

```swift
enum VideoPreviewStatus: String, Codable {
    case pending
    case ready
    case failed
}
```

### 为什么第一版优先本地首图

因为当前上传流程本来就先拿到本地临时视频文件，再上传 OSS。

这意味着我们可以在上传成功后，立刻基于本地文件生成一张首帧图并缓存到本地，然后把缓存路径写回 `TrainingVideoPost`。这样有几个优势：

- 用户上传完成后不用等远程缩略图服务
- 首图生成逻辑可以完全在客户端闭环
- 不需要新增后端接口
- 与当前“先打通本地主链路”的策略一致

## 首图生成与缓存方案

### 生成时机

推荐在上传成功、写入 `TrainingVideoPost` 之前或同时生成首图。

主流程变成：

`选择视频 -> 上传 OSS -> 读取 remoteURL -> 生成本地缩略图 -> 写入 TrainingVideoPost -> 动态流刷新`

### 生成方式

建议新增一个轻量服务，例如：

- `VideoPreviewService`

职责：

- 从本地视频 URL 提取首帧图
- 读取视频时长
- 将缩略图保存到 App 可访问的缓存目录
- 返回缩略图路径和时长信息

建议技术路线：

- 使用 `AVAssetImageGenerator` 生成首帧
- 使用 `AVURLAsset` 读取时长
- 缩略图文件保存到 `Application Support` 或 `Caches`

### 缓存策略

第一版建议采用简单可控策略：

- 以 `post.id` 作为文件名
- 使用 JPEG 存储
- 单张尺寸控制在足够卡片显示的范围内
- 不做复杂清理策略，后续再补

推荐目录：

- `Library/Caches/VideoPreviews/`

原因：

- 这类文件是可再生资源
- 被系统清理后仍可重新生成或降级显示

### 降级策略

如果首图生成失败：

- `previewStatus = .failed`
- 卡片首图区域显示统一占位样式
- 仍保留播放 icon、文件名和其他元信息
- 不阻断 `TrainingVideoPost` 写入

首图生成失败不能影响上传主链路成功。

## 播放方案

### 详情页播放源优先级

建议按下面顺序选择播放资源：

1. `remoteVideoURLString`
2. 未来可能的本地持久化视频路径
3. 无可用地址时展示不可播放态

第一版不建议依赖上传后的临时文件继续播放，因为当前上传完成后会删除临时文件，这与现有实现一致，也更节省空间。

### 播放组件

建议在详情页使用系统视频播放器能力。

推荐方向：

- SwiftUI 的 `VideoPlayer`
- 或包装 `AVPlayerViewController`

选择原则：

- 第一版优先简单集成
- 先满足可播放，再决定是否要更细控制

### 自动播放策略

第一版建议不要进入详情页就自动播放。

原因：

- 用户可能只是想看信息，不一定立刻播放
- 自动播放会增加网络请求和页面抖动
- 和“首图 + 点按播放”的心智更一致

默认行为：

- 详情页先显示封面
- 用户点击播放按钮后开始播放

## 风险与取舍

### 1. 远程地址未必长期稳定

如果 OSS `publicURL` 未来改成需要签名，`remoteVideoURLString` 可能不能长期直连。

应对策略：

- 第一版先复用当前 `publicURL`
- 模型层保留 `objectKey`
- 后续如需签名播放，再补“按 objectKey 动态换取可播地址”的服务

### 2. 首图生成耗时影响上传完成感知

如果把首图生成放在主线程或和上传成功强绑定，可能让“上传完成”反馈变慢。

应对策略：

- 首图生成必须在后台处理
- `TrainingVideoPost` 可以先以 `previewStatus = .pending` 写入
- 生成完成后再回写路径和状态

### 3. 列表图片加载带来滚动开销

如果卡片首图直接加载原尺寸图片，滚动会变差。

应对策略：

- 缩略图落盘时就控制尺寸
- 卡片只显示压缩后的本地图片

### 4. 旧数据没有首图

已有 `TrainingVideoPost` 或 legacy 迁移数据不会自动拥有首图。

应对策略：

- 允许旧数据显示占位图
- 后续可选做后台补生成

## 验证建议

实现完成后，至少验证以下场景：

1. 新上传训练视频后，动态页立即显示首图
2. 新上传比赛视频后，动态页立即显示首图
3. 首图生成失败时，卡片仍正常出现
4. 点击卡片进入详情页后可以播放
5. 无远程地址时，详情页显示不可播放态
6. 家长端查看已有视频时，卡片结构与球员端一致，只增加成员信息
7. 旧数据迁移后，没有首图的视频仍能稳定展示

如涉及真实界面验收，优先使用 `scripts/build-and-launch.sh` 做端到端验证，并补截图确认。

## 结论

这次补的不是一个“封面图小功能”，而是让视频动态真正成为视频内容流。

推荐按下面顺序实施：

1. 先补 `TrainingVideoPost` 的预览与播放字段
2. 再接上传后的本地首图生成
3. 让动态卡片先具备“看见视频”的能力
4. 最后把详情页升级成可播放的视频详情页

这样改动范围可控，也最符合当前仓库“先把主链路跑通，再逐步增强”的节奏。
